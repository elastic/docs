Manage Security Certificates

You install your own TLS/SSL certificates so that you can connect securely to Elastic Cloud Enterprise using the HTTPS protocol. Without these added certificates, your browser returns a security warning when you try to connect to the Cloud UI over HTTPS on port 12443, for example.

In these instructions, we show you how you can download the security certificate that gets generated during the Elastic Cloud Enterprise install and use it to add your own TLS/SSL certificates. You can add your TLS/SSL certificates any time after you have installed Elastic Cloud Enterprise on your hosts. In addition to the steps shown here, you might also need to import your CA certificate into your browser certificate chain, if you don’t already use the same certificate within your organization.

You need to change the certificates for the following three Elastic Cloud Enterprise components:

Proxy
Requires a certificate to connect securely to Kibana, for example. Requires a wildcard certificate.
Admin console
Requires a certificate for secure RESTful API calls.
Cloud UI
Requires a certificate to connect securely to the Cloud UI.

After the certificates have been installed, connecting securely to Kibana and the Cloud UI or making secure RESTful API calls should not result in any security warnings or errors.

Before you begin:

  1. If your organization already uses a CA certificate and X.509 certificate chain, you need to have these files ready. Alternatively, you can generate a CA certificate and X.509 certificate chain.

  2. Obtain the existing certificate generated during the installation of Elastic Cloud Enterprise, where CLUSTER_ID and LOCAL_HOST_IP are values that depend on your specific installation (bc898abb421843918ebc31a513169a28.54.152.125.221.ip.es.io:12443, for example): 

    openssl s_client -showcerts -connect CLUSTER_ID.HOST_IP.ip.es.io:12443 < /dev/zero

Generate a CA Certificate and X.509 Certificate Chain

The steps in this section provide high-level instructions on what you need to do if you do not already have a CA certificate and X.509 certificate chain. The method by which you generate the certificate and certificate chain differs by operating system, and the exact steps are outside the scope of these instructions.

To generate the necessary files:

  1. Generate a certificate authority (CA) RSA key pair.
  2. Create a self-signed CA certificate.
  3. Generate a server RSA key pair.
  4. Create a certificate signing request (CSR) for server certificate with the common name and the alternative name set.
  5. Sign the server CSR with CA key pair.
  6. Concatenate the PEM encode server RSA private key, the server certificate, and the CA certificate into a single file.

In subsequent steps, the concatenated file containing the RSA private key, server certificate, and CA certificate is referred to as the CA_CERTIFICATE_FILENAME and used to add your own TLS/SSL certificates.

Proxy

To change the proxy certificate:

  1. Add the wildcard certificate chain for the proxy to your Elastic Cloud Enterprise installation, where CA_CERTIFICATE_FILENAME is the name of the CA certificate you downloaded earlier and PROXY_PEM_FILENAME is the name of the concatenated file containing the RSA private key, server certificate, and CA certificate: 

    curl -vvv --cacert CA_CERTIFICATE_FILENAME -H 'Content-Type: application/json' --data-binary @PROXY_PEM_FILENAME --user "root:PASSWORD" "https://admin:12343/api/v1/platform/tls/proxy"
  2. Log out of any Kibana instances you might be logged into and log in again.

  3. Verify that you are now using the new certificate chain. There should be no security warnings when you connect to the Kibana endpoint over HTTPS in your web browser.

    Alternatively, you can also use the openssl command to check the proxy certificates:

    openssl s_client -CAfile CA_CERTIFICATE_FILENAME -showcerts -connect containerhost:9243 < /dev/zero
openssl s_client -CAfile CA_CERTIFICATE_FILENAME -showcerts -connect containerhost:9343 < /dev/zero

Admin Console

To change the admin console certificate:

  1. Add the certificate for the admin console to your Elastic Cloud Enterprise installation, where CA_CERTIFICATE_FILENAME is the name of the CA certificate you downloaded earlier and ADMIN_PEM_FILENAME is the name of the concatenated file containing the RSA private key, server certificate, and CA certificate: 

    curl -vvv -H 'Content-Type: application/json' --data-binary @ADMIN_PEM_FILENAME --user "root:PASSWORD" "http://containerhost:12300/api/v1/platform/tls/adminconsole"`

  2. You should see the certificate change for the admin console when you connect over HTTPS in your web browser, but you can also check the certificate using the openssl command: 

    openssl s_client -CAfile CA_CERTIFICATE_FILENAME -showcerts -connect containerhost:12343 < /dev/zero

    If successful, the output includes verify return:1:

    depth=1 C = US, ST = California, L = Mountain View, O = "Elasticsearch, Inc.", CN = ECE Certificate Authority
verify return:1
depth=0 C = US, ST = California, L = Mountain View, O = "Elasticsearch, Inc.", CN = admin
verify return:1

    If unsuccessful, the output includes a verify error: indicator.

  3. To use the new certificate when making RESTful API calls, include the --cacert CA_CERTIFICATE_FILENAME parameter, where CA_CERTIFICATE_FILENAME is now the name of your own CA certificate, and change the protocol and port to https://containerhost:12343.

  4. Update your /etc/hosts to add admin and admin-update as an alias for your host: 

    sudo vim /etc/hosts
192.168.43.10 containerhost   admin   admin-update

Cloud UI

To change the Cloud UI certificate:

  1. Add the certificate for the Cloud UI to your Elastic Cloud Enterprise installation, where CA_CERTIFICATE_FILENAME is the name of the CA certificate you downloaded earlier and CLOUDUI_PEM_FILENAME is the name of the concatenated file containing the RSA private key, server certificate, and CA certificate: 

    curl -vvv --cacert CA_CERTIFICATE_FILENAME -H 'Content-Type: application/json' --data-binary @CLOUDUI_PEM_FILENAME --user "root:PASSWORD" "http://admin:12300/api/v1/platform/tls/ui"
  2. Log out of the Cloud UI and log in again.

  3. Verify that you are now using the new certificate chain. There should be no security warnings when you connect to the Cloud UI over HTTPS in your web browser.

    Alternatively, you can also check the certificate using the openssl command:

    openssl s_client -CAfile CA_CERTIFICATE_FILENAME -showcerts -connect containerhost:12443 < /dev/zero
